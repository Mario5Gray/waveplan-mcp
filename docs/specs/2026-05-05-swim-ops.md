# SWIM Operations Manual

Operator-facing reference for `waveplan-cli swim`. For design context see [`2026-05-05-swim-implementation-plan.md`](2026-05-05-swim-implementation-plan.md).

> **v1 scope.** Wave 4 (CLI + this doc) is the production surface. Wave 5 (MCP swim tools, refinement compiler) is pending; commands marked `out of scope` below ship in a later phase.

---

## Quick start

Three commands take a plan from authored to first-step-applied:

```bash
# 1) Compile the schedule from a plan + agents config.
waveplan-cli swim compile-schedule \
  --plan docs/specs/swim-ops-examples/plan.json \
  --agents docs/specs/swim-ops-examples/waveagents.json \
  --out /tmp/schedule.json

# 2) Resolve the next safe step (read-only).
waveplan-cli swim next --schedule /tmp/schedule.json

# 3) Apply that step (mutates state + journal under .waveplan/swim/<plan>/).
waveplan-cli swim step --apply --schedule /tmp/schedule.json
```

Every subcommand emits JSON on stdout. Exit codes: `0` ok, `2` usage, `3` runtime error, `4` `unknown_pending` (actionable).

## Subcommand reference

### `swim compile-schedule`

Generates a v2 schedule from a plan + agents config by shelling to `wp-emit-wave-execution.sh`.

```
swim compile-schedule --plan <plan.json> --agents <waveagents.json>
                      [--out <schedule.json>] [--task-scope all|open]
```

| flag | required | meaning |
|---|---|---|
| `--plan` | yes | path to `*-execution-waves.json` |
| `--agents` | yes | agents config (`{agents: [{name,provider}], schedule: [...]}`) |
| `--out` | no | write to file; otherwise stdout |
| `--task-scope` | no | `all` (default) or `open` (only un-completed tasks) |

Output: schedule JSON conforming to [`swim-schedule-schema-v2.json`](swim-schedule-schema-v2.json).

### `swim next`

Read-only: returns the next safe step without mutation. Calls `ResolveNext` over the live schedule + journal + state.

```
swim next --schedule <schedule.json> [--journal <journal.json>] [--state <state.json>]
```

Output: `Decision` JSON with one of `action: ready | blocked | done | drift`.

```json
{
  "action": "ready",
  "code": "ready",
  "cursor": 0,
  "row": { "step_id": "S1_T1.1_implement", "seq": 1, "task_id": "T1.1", "action": "implement", ... }
}
```

`action: drift` indicates the live state has advanced past the journal cursor â€” see [Cursor drift](#cursor-drift).

### `swim step`

Single-step operations: resolve, apply, or ack-unknown.

```
swim step --schedule <schedule.json> [--journal <journal.json>] [--state <state.json>]
          [--seq N | --step-id ID]
          [--apply] [--ack-unknown <step_id> --as failed|waived]
```

Mutual exclusions enforced:
- `--apply` âŠĄ `--ack-unknown`
- `--seq` âŠĄ `--step-id`

Without `--apply` or `--ack-unknown`, `step` is equivalent to `next` (read-only).

With `--apply`, runs the locked A/B/C transaction (acquire lock â†’ snapshot A â†’ snapshot B same-token check â†’ invoke argv â†’ snapshot C postcondition check â†’ append `applied` event + cursor++). Output is `ApplyReport`:

```json
{
  "status": "applied",
  "step_id": "S1_T1.1_implement",
  "seq": 1,
  "exit_code": 0,
  "stdout_path": ".waveplan/swim/swim-ops-example/logs/S1_T1.1_implement.1.stdout.log",
  "stderr_path": ".waveplan/swim/swim-ops-example/logs/S1_T1.1_implement.1.stderr.log",
  "reason": "",
  "hint": ""
}
```

`status` is one of: `applied | blocked | done | incomplete_dispatch | lock_busy | unknown_pending`. See [JSON contract glossary](#json-contract-glossary).

### `swim run`

Loop `Apply` until a stop condition fires.

```
swim run --schedule <schedule.json> [--journal <journal.json>] [--state <state.json>]
         --until <implement|review|fix|end_review|finish|seq:N|step:ID>
         [--dry-run] [--max-steps N]
```

Stop conditions:
- bare action name (`implement`/`review`/`fix`/`end_review`/`finish`): stop after first `applied` step matching that action
- `seq:N`: stop after seq=N applies (inclusive)
- `step:S1_T1.1_finish`: stop after that specific step applies
- empty: stop on first non-`applied` report (default)

`--dry-run` resolves the path without mutation, lock, or log files. Emits `would_apply` / `would_block` per simulated step.

`--max-steps N` caps the loop (0 = unbounded).

Output: `RunReport` containing the full sequence of `ApplyReport`s plus `stopped` reason and `reached_until` flag.

### `swim journal`

Read-only journal inspection.

```
swim journal --schedule <schedule.json> [--journal <journal.json>] [--tail N]
```

`--tail N` returns the last N events. Default: full journal.

Output: journal JSON conforming to [`swim-journal-schema-v1.json`](swim-journal-schema-v1.json).

### `swim validate`

Validates a schedule, journal, or plan against its published schema.

```
swim validate --kind schedule|journal|plan --in <path>
```

Output: `{"ok": true}` on pass; `{"ok": false, "error": "..."}` with exit 3 on fail.

### `swim compile-plan-json`

Safe canonicalizer for execution-waves plan files. Validates structure + ref-integrity + types; preserves all unknown-but-valid fields.

```
swim compile-plan-json --in <plan.json> [--out <canon.json>]
```

Independent of the runtime path. Useful for normalizing plans before commit.

## Recovery flows

### Stuck `unknown` after crash

A process crashed mid-apply between invoke and journal-write. Next read promotes the orphan to `outcome: unknown`, which blocks the cursor.

```bash
# 1) See what's blocking.
waveplan-cli swim journal --schedule /tmp/schedule.json --tail 3
# Output shows event with started_on set and outcome="unknown".

# 2) Decide: did the invoked command actually succeed?
#    Inspect the log files referenced in stdout_path / stderr_path.
cat .waveplan/swim/swim-ops-example/logs/S1_T1.1_implement.1.stdout.log
cat .waveplan/swim/swim-ops-example/logs/S1_T1.1_implement.1.stderr.log

# 3a) If the command succeeded externally (e.g. file changes are present):
#     waive â€” cursor advances past it.
waveplan-cli swim step --schedule /tmp/schedule.json \
  --ack-unknown S1_T1.1_implement --as waived

# 3b) If the command did NOT succeed: mark failed (retry-eligible).
waveplan-cli swim step --schedule /tmp/schedule.json \
  --ack-unknown S1_T1.1_implement --as failed

# 4) Confirm cursor moved (or is now ready to retry).
waveplan-cli swim next --schedule /tmp/schedule.json
```

`--as` is required: ack is destructive enough to demand an explicit outcome.

### `lock_busy` from another agent

Another swim process holds `.waveplan/swim/<plan>/swim.lock`.

```bash
waveplan-cli swim step --apply --schedule /tmp/schedule.json
# Exit 3, JSON:
# {
#   "status": "lock_busy",
#   "hint": "{\"pid\":47215,\"started_at\":\"2026-05-09T14:22:01Z\",\"hostname\":\"ops-mac\"}"
# }

# Identify the holder.
ps -p 47215 -o pid,ppid,user,etime,command 2>/dev/null || echo "pid no longer running"

# Two paths:
#   (a) Coordinate â€” wait for them to finish.
#   (b) If pid is dead, the lock will be released by next file-system probe;
#       if it isn't (NFS / stale advisory lock), remove the lock file manually:
rm .waveplan/swim/swim-ops-example/swim.lock

# Re-attempt.
waveplan-cli swim step --apply --schedule /tmp/schedule.json
```

Manual lock removal is safe only when you are certain the holder is gone â€” flock is advisory.

### Cursor drift

`swim next` returns `action: drift` / `code: cursor_drift` when the live state has advanced past the journal cursor (e.g. an out-of-band script directly mutated state).

```bash
waveplan-cli swim next --schedule /tmp/schedule.json
# {
#   "action": "drift",
#   "code": "cursor_drift",
#   "cursor": 0,
#   "reason": "cursor_drift: action=implement requires=available actual=taken (state ahead)"
# }
```

**v1 recovery is manual.** Edit the journal cursor by hand to the row whose preconditions match the live state, then re-run `swim next` to confirm `action: ready`.

```bash
# Find the journal path (default: <schedule>.journal.json).
JOURNAL=$(dirname /tmp/schedule.json)/$(basename /tmp/schedule.json .json).journal.json

# Cursor is an integer index into execution[]. Increment manually.
jq '.cursor = 1' "$JOURNAL" > /tmp/journal.new && mv /tmp/journal.new "$JOURNAL"

# Confirm.
waveplan-cli swim next --schedule /tmp/schedule.json
```

## Safe `--apply` checklist

Run before every production `swim step --apply`:

1. **Plan + state pinned**: `swim validate --kind schedule --in <schedule.json>` returns `{"ok": true}`.
2. **No pending `unknown`**: `swim journal --tail 5` shows no orphan events. If any exist, ack them first.
3. **No competing lock**: `cat .waveplan/swim/<plan>/swim.lock` is empty or stale-pid only.
4. **Dry-run preview**: `swim run --until <next-action> --dry-run` shows the expected sequence with no `would_block`.
5. **Have rollback plan**: know which file to revert (state JSON) and which logs to inspect (under `.waveplan/swim/<plan>/logs/`).

## Boundaries and Inquiries

SWIM observes execution and stops at expected boundaries; it does not approve downstream prompts or grant target-CLI permissions. `--until review` means "apply safe rows until the review boundary," not "answer permission prompts on the way." Permission handling lives in adapters (`wp-task-to-agent.sh`, target CLIs), not in SWIM.

### Boundary classes

Each `ApplyReport` has a `boundary` field that classifies *why* the row stopped. `RunReport.boundary` rolls up the last step's boundary.

| boundary | meaning |
|---|---|
| `until_reached` | the configured `--until` condition fired on an `applied` row |
| `done` | the schedule is complete; no row to apply |
| `max_steps` | `--max-steps N` guardrail reached before another boundary |
| `blocked` | precondition, postcondition, drift, lock, or invoke failure |
| `incomplete_dispatch` | runtime state advanced but the dispatch receipt is missing |
| `unknown_pending` | an in-flight event was interrupted and must be ack'd |

`boundary` mirrors `status` one-to-one for non-applied rows: `status: blocked` carries `boundary: blocked`; `status: incomplete_dispatch` carries `boundary: incomplete_dispatch`. `applied` rows have no `boundary` (the field is omitted).

### Inquiry decorators

When a row stops because the target CLI is waiting on a human or agent (permission prompt, sandbox approval, missing credential), SWIM decorates the report with three optional fields:

| field | type | meaning |
|---|---|---|
| `inquiry_required` | bool | true when a downstream party must answer before progress is possible |
| `inquiry_source` | string | `adapter` \| `invoke` \| `timeout` \| `unknown` |
| `inquiry_hint` | string | free-text actionable hint (e.g., "answer in TUI, or set non-interactive policy") |

These fields are decorators, not a separate boundary class. They can attach to any `blocked` or `incomplete_dispatch` row. Default state is omitted; consumers must treat absence as "no inquiry signal," not "no inquiry."

### Detection sources (in trust order)

SWIM derives `inquiry_*` from these sources, preferring the most trustworthy:

1. **Dispatch receipt fields.** The adapter writes `inquiry_required`, `inquiry_source`, and `inquiry_hint` into `<step>.<attempt>.dispatch.json`. This is the canonical signal because the adapter knows the target CLI's state directly. Implemented.
2. **Explicit env-var / file contract.** Used only when extending the receipt isn't feasible. Not implemented.
3. **Stdout/stderr heuristics.** Last resort, gated by config; off by default. Not implemented.

Today only the receipt path is wired. SWIM reads the receipt matching the exact journal event attempt for dispatch actions (`implement`, `review`, `fix`) and copies the inquiry fields through to the report.

### Adapter receipt contract

Adapters writing the dispatch receipt may include inquiry signals:

```json
{
  "ok": true,
  "inquiry_required": true,
  "inquiry_source": "adapter",
  "inquiry_hint": "target CLI requested permission; answer in TUI or configure non-interactive policy"
}
```

Fields are all optional. `inquiry_required: false` (or omitted) means SWIM emits no decorator on the report.

### Operator implications

- `--until` is still an execution boundary. A row blocked on a permission prompt halts the run with `reached_until: false`; it does not silently skip.
- `RunReport.stopped` remains a coarse rollup label and continues to take values like `until` / `non_applied` / `max_steps`. Prefer `boundary` for finer classification.
- Adapters own the inquiry contract. SWIM never prompts and never answers; it only reports.

## JSON contract glossary

### `Decision` (output of `swim next`)

| field | type | meaning |
|---|---|---|
| `action` | string | `ready` \| `blocked` \| `done` \| `drift` |
| `code` | string | `ready` \| `blocked_precondition` \| `cursor_drift` \| `done` \| `bad_cursor` \| ... |
| `cursor` | int | journal cursor position |
| `row` | object | the schedule row at cursor (when applicable) |
| `reason` | string | human-readable explanation |

### `ApplyReport` (output of `swim step --apply`)

| field | type | meaning |
|---|---|---|
| `status` | string | `applied` \| `blocked` \| `done` \| `incomplete_dispatch` \| `lock_busy` \| `unknown_pending` |
| `step_id` | string | the step the apply targeted |
| `seq` | int | sequence number |
| `exit_code` | int | invoke exit code (0 if not invoked) |
| `stdout_path` | string | absolute path to captured stdout (empty if not invoked) |
| `stderr_path` | string | absolute path to captured stderr |
| `reason` | string | one of `missing_last_event` \| `precondition_unmet` \| `race_detected` \| `postcondition_mismatch` \| invoke-failure detail |
| `hint` | string | actionable diagnostic â€” JSON for `lock_busy` (holder pid/started_at/hostname); step_id for `unknown_pending` |
| `boundary` | string | one of `until_reached` \| `done` \| `max_steps` \| `blocked` \| `incomplete_dispatch` \| `unknown_pending`. Omitted on `applied` rows. See [Boundaries and Inquiries](#boundaries-and-inquiries). |
| `inquiry_required` | bool | decorator. Omitted unless a downstream party must answer before progress is possible. |
| `inquiry_source` | string | decorator. `adapter` \| `invoke` \| `timeout` \| `unknown`. |
| `inquiry_hint` | string | decorator. Free-text actionable hint. |

### `RunReport` (output of `swim run`)

| field | type | meaning |
|---|---|---|
| `steps` | array of `ApplyReport` | one entry per step attempted |
| `stopped` | string | `until` \| `non_applied` \| `max_steps` \| `lock_busy` \| ... |
| `reached_until` | bool | true iff the `--until` condition fired |
| `dry_run` | bool | true iff `--dry-run` was set |
| `boundary` | string | coarse rollup. Set to the terminal step's `boundary` on a non-applied stop, `until_reached` when `--until` fires, `max_steps` on the guardrail, `done` when the schedule is exhausted (wet or dry-run). |
| `inquiry_required` | bool | rollup of the last step's `inquiry_required`. |
| `inquiry_source` | string | rollup of the last step's `inquiry_source`. |
| `inquiry_hint` | string | rollup of the last step's `inquiry_hint`. |

### Journal `event` (entries inside the journal file)

| field | type | meaning |
|---|---|---|
| `event_id` | string | `E0001` etc., monotonic per journal |
| `step_id` | string | matching schedule row |
| `seq` | int | matching schedule seq |
| `task_id` | string | parent task |
| `action` | string | one of `implement` \| `review` \| `fix` \| `end_review` \| `finish` |
| `attempt` | int | 1-indexed retry count |
| `started_on` | string | ISO 8601 |
| `completed_on` | string | ISO 8601 (omitted for in-flight) |
| `outcome` | string | `applied` \| `failed` \| `blocked` \| `unknown` \| `waived` (omitted for in-flight) |
| `exit_code` | int | invoke exit code |
| `stdout_path` | string | log file path |
| `stderr_path` | string | log file path |
| `state_before` | object | `{task_status: ...}` |
| `state_after` | object | `{task_status: ...}` |

In-flight events have `started_on` set but `completed_on` and `outcome` absent â€” this is the orphan signature recovery scans for.

## Exit codes

| code | meaning | example |
|---|---|---|
| 0 | success | normal apply, valid schedule, ready next-step |
| 2 | usage error â€” bad arguments | unknown subcommand, mutex violation, missing required flag |
| 3 | runtime error â€” operation could not complete | `lock_busy`, schema validation failure, file not found, plan-state mismatch |
| 4 | `unknown_pending` â€” actionable | journal has orphan events; operator must run `step --ack-unknown` before cursor can advance |

`4` is split out from `3` because automation can branch on it: an orchestrator seeing exit `4` knows precisely what to do next, whereas `3` is a generic stop-and-inspect.
